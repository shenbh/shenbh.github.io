内存抖动

概念：在循环中创建对象（比如onDraw中创建对象）内存达到临界值，会触发GC。频繁创建，频繁触发GC。从内存分析器（如AS–Run–Profiler（Memory Profiler））内看就是忽上忽下的波动，称作Memory Churn（内存抖动）。

OOM

ANR

产生 ANR 的原因

当前的事件没有机会得到处理，例如 UI 线程正在响应另一事件，当前事件由于某种原因被阻塞了。

当前的事件正在处理，但由于耗时太长没能及时完成。

ANR 种类（原因）

ANR 产生原因（对应组件） 超时时间 备注
Activity/View KeyDispatchTimeout（5 秒内） 主要类型。按键或触摸事件在特定时间内无响应
BroadcastReceiver BroadcastTimeout（10 秒内） BroadcastReceiver在特定时间内无法处理完成
Service ServiceTimeout（20 秒内） 小概率类型。Service在特定事件内无法处理完成

典型 ANR 问题场景

UI 线程中存在耗时操作，如在 UI 线程中进行网络请求（在 Android4.0 之后会抛异常）、数据库操作、文件操作、复杂逻辑计算等，导致 UI 线程无法及时处理用户输入等。（要放到子线程中进行，然后通过handler.sendMessage、runonUIThread\AsyncTask等方式更新UI）

UI 线程等待子线程释放某个锁，从而无法处理用户输入。

耗时的动画需要大量的计算工作，可能导致 CPU 负载过重。

anr 记录文件 traces.txt 分析

数据为《Android 高级进阶》-顾浩鑫p138

  • 发生 ANR 的进程名称、ID、时间

  • 手机的 CPU 架构：arm

  • 堆内存信息：24% free，16MB/21MB; 53963 objects

  • 主线程基本信息

    • 线程名称：”main“
    • 线程优先级：prio=5
    • 线程锁 ID：tid=1
    • 线程状态：Sleeping

  • 主线程的详细信息

    • 线程组名称：group=”main“
    • 线程被挂起的次数：sCount=1
    • 线程被调试器挂起的次数：dsCount=0，当线程被调试结束后，sCount 会被重置为 0，它的值会重新根据是否被挂起而增加，而 dsCount 不会被重置为 0.因此，dsCount 可以用来判断这个线程是否被调试器调试过。
    • 线程的 Java 对象地址：obj=0x87789ef0
    • 线程本身的 Native 对象地址：self=0xb4f07800

  • 线程的调度信息

    • LInux 系统中内存线程 ID：sysTid=8672，可以看到主线程的线程号和进程号相同
    • 线程调度优先级：nice=0
    • 线程调度组：cgrp=apps
    • 线程调度策略和优先级：sched=0/0
    • 线程处理函数地址：handle=0xb6fe4ec8

  • 线程的上下文信息

    • 线程调度状态：state=S
    • 线程在 CPU 中的执行时间、线程等待时间、线程执行的时间片长度：schedstat=（246636720 148938849 453）
    • 线程在用户态中调度时间值（单位：jiffies）：utm=18
    • 线程在内核态中的调度时间值：stm=6
    • 最后执行这个线程的 CPU 核序号：core=3

  • 线程的堆栈信息

    • 堆栈地址和大小：stack=0xbe5bd000-0xbe5bf000 stackSize=8MB
    • 堆栈信息：从中可以看到，ANR 是由于在 MainActivity 类中的 triggerAnrWithLongOperation 函数调用了 Thread.sleep 导致

ANR 的避免和检测

StrictMode严格模式

主要监测两大类问题：

  • 线程策略 ThreadPolicy
    • detectCustomSlowCalls：监测自定义耗时操作
    • detectDiskReads：监测是否存在磁盘读取操作
    • detectDiskWrites：监测是否存在磁盘写入操作
    • detectNetwork：监测是否存在网络操作
  • 虚拟机策略 VmPolicy
    • detectActivityLeaks：监测是否存在 Activity 泄露
    • detectLeakedClosableObjects：监测是否存在未关闭的 Closable 对象泄露
    • detectLeakedSqlLiteObjects：监测是否存在 Sqlite 对象泄露
    • setClassInstanceLimit：监测类实例个数是否超过限制

在应用初始化的地方）Application 或 MainActivity 的 onCreate 中执行如下：

penaltyLog()：在 logcat 中打印日志

detectAll()：开启所有检测策略。若只开启某些策略，则 detectAll()替换成它们（比如 .detectDiskWrites().detectNetwork()）

BlockCanary

一个非侵入式的性能监控函数库，用法和 LeakCanary 类似，只不过后者监控应用内存泄露，而 BlockCanary 主要用来监控应用主线程的卡顿。基本原理是利用主线程的消息队列处理机制，通过对比消息分发开始和结束的时间点来判断是否超过设定的时间，如果是，则判断主线程卡顿。
